# 设计：享元模式——特工享元池

## 上下文

当前项目是一个 Java 设计模式学习项目，每个模式都通过生动的业务场景来演示其核心思想。享元模式（Flyweight Pattern）的目标是在大量细粒度对象的场景中，通过共享内部状态来减少内存占用。项目已有观察者、策略、装饰器等模式，但缺少结构型模式中的享元模式。为了与项目整体风格保持一致，本次设计采用「情报组织特工复用」作为业务隐喻。

## 目标 / 非目标

**目标：**
- 演示享元模式的核心结构：享元接口、具体享元、享元工厂、外部状态上下文
- 通过控制台输出生动展示「大量任务 vs 少量对象」的内存复用效果
- 保持与现有项目一致的代码风格和包结构

**非目标：**
- 不引入真实的内存分析工具（如 JMX、VisualVM）
- 不涉及线程安全或并发控制（作为基础教学示例，保持简单）
- 不修改任何现有模式模块的代码

## 决策

### 1. 外部状态使用不可变值对象 `MissionContext`
**选择：** 将所有外部状态（location, target, equipment）封装在一个独立的 `MissionContext` 类中，作为 `Agent.executeMission()` 的参数。
**理由：** 这可以清晰地区分内部状态与外部状态，避免具体享元对象被意外修改，也更符合享元模式的最佳实践。
**替代方案：** 将外部状态拆分为多个方法参数 → 会导致接口冗长，不利于扩展。

### 2. 享元工厂使用 `HashMap<String, Agent>` 管理池
**选择：** `AgentFactory` 内部使用 `Map<String, Agent>` 缓存已创建的特工对象，按 `codeName` 作为 key。
**理由：** 实现简单直观，适合教学演示。特工种类有限（3 种），Map 的查找效率足够。
**替代方案：** 使用 `ConcurrentHashMap` 或带锁的机制 → 对于当前单线程演示场景过度设计。

### 3. 特工种类固定为 3 个：夜莺、猎豹、幽灵
**选择：** 在 `AgentFactory` 中预定义或通过懒加载创建 3 种特工享元。
**理由：** 3 个对象 vs 100 个任务的对比效果足够震撼，也便于控制台输出展示。
**替代方案：** 通过配置文件或枚举动态扩展 → 增加了不必要的复杂度，偏离教学重点。

### 4. `Main.java` 同时承担客户端和演示入口职责
**选择：** `Main.java` 既作为 `MissionControl` 的调用方，也直接输出总结信息。
**理由：** 与项目中其他模式模块（如 `strategy.Main`、`observer.Main`）保持一致，都是单文件入口 + 控制台日志的演示方式。

## 风险 / 权衡

- **[风险] 外部状态传递错误可能导致特工「串台」** → **缓解：** 通过 `MissionContext` 不可变对象封装，客户端每次调用都传入新的上下文实例。
- **[风险] 控制台输出过多（100 条任务日志）可能刷屏** → **缓解：** 仅打印前 5 条和最后 1 条任务日志作为代表，中间用省略号表示。
