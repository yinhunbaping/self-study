## ADDED Requirements

### 需求:特工享元池必须管理有限数量的特工对象
系统必须仅创建少量特工对象，并通过享元工厂统一管理其生命周期，避免为每个任务都新建特工实例。

#### 场景:同一特工执行多个任务
- **当** 情报总部向不同地点派发多个任务
- **那么** 系统必须复用已有的特工对象，而非每次都创建新实例

### 需求:特工对象必须区分内部状态与外部状态
特工的内部状态（代号、外貌特征、专业技能）必须在创建后保持不变；外部状态（任务地点、行动目标、携带装备）必须由客户端在每次执行任务时传入。

#### 场景:特工复用时切换任务上下文
- **当** 客户端调用特工执行任务并传入 `MissionContext`
- **那么** 特工必须根据传入的外部状态输出对应的任务执行日志
- **且** 特工的内部状态在执行过程中不得被修改

### 需求:客户端必须能够批量派发任务并观察内存复用效果
客户端 `MissionControl` 必须支持批量创建任务请求，并在所有任务执行完毕后输出总结信息，包括总任务数和实际创建的特工对象数。

#### 场景:批量派发 100 个任务
- **当** 客户端派发 100 个任务请求
- **那么** 系统仅创建 3 个特工对象用于处理所有任务
- **且** 控制台输出总结信息展示内存复用比例
